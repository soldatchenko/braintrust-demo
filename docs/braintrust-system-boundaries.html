System Boundary Diagrams

Where each step executes, which system owns it, and where the API calls cross boundaries.

Diagram 1 — Local Development Workflow
Your machine orchestrates everything. Cloud services are called via API.
Phase A — Document Ingestion (run once or on doc changes)
# Step Runs In Calls Out To What Happens
1 Load documents YOUR MACHINE local filesystem Python reads markdown/text files from a local directory (or pulls from S3 via boto3)
2 Chunk documents YOUR MACHINE nothing — pure Python Your chunker function splits each doc by heading or token count. Outputs a list of {text, metadata} objects. No API calls.
3 Generate embeddings YOUR MACHINE OpenAI API For each chunk, call openai.embeddings.create() with text-embedding-3-small. Runs locally, sends text to OpenAI, gets back 1536-dim vectors.
4 Upsert vectors YOUR MACHINE Pinecone API Call index.upsert() sending vectors + metadata. Pinecone stores and indexes them. Your machine just sends the HTTP request.
Phase B — RAG Query Loop (iterative development)
# Step Runs In Calls Out To What Happens
5 Embed the query YOUR MACHINE OpenAI API Same embedding call as step 3, but for the user's question. Returns a query vector.
6 Retrieve context YOUR MACHINE Pinecone API Call index.query(vector, top_k=5). Pinecone runs similarity search and returns matching chunks with metadata.
7 Assemble prompt YOUR MACHINE nothing — string formatting Stitch retrieved chunks into a system prompt template with the user query. Pure Python string operations.
8 Generate answer YOUR MACHINE OpenAI API
Braintrust Proxy		 Call client.chat.completions.create() via wrap_openai. Request routes through Braintrust AI Proxy → OpenAI. Braintrust auto-captures the trace span.
Phase C — Evaluation & Scoring (run per experiment)
# Step Runs In Calls Out To What Happens
9 braintrust eval YOUR MACHINE Braintrust API CLI loads your eval file, pulls dataset from Braintrust, creates an experiment. Orchestration runs locally.
10 Execute task × N YOUR MACHINE steps 5-8 repeat For each test case in the dataset, the eval framework calls your RAG function. All the same API calls happen per test case.
11 Score outputs YOUR MACHINE OpenAI API Autoevals scorers run locally. Deterministic scorers (Levenshtein) are pure computation. LLM-as-judge scorers (Factuality, Faithfulness) call OpenAI to grade each output.
12 Publish experiment YOUR MACHINE Braintrust API Scores, traces, and outputs are pushed to Braintrust. You view results in the Braintrust UI (browser).
Phase D — Iterate (playground + prompt tweaks)
# Step Runs In Calls Out To What Happens
13 Review in UI BRAINTRUST UI browser Compare experiments side-by-side, inspect per-test-case diffs, identify regressions. This is the one step that doesn't run on your machine.
14 Playground prototyping BRAINTRUST UI OpenAI API Braintrust's Playground runs prompts server-side against your datasets. Useful for quick prompt iteration without touching code.
15 Update code, re-eval YOUR MACHINE back to step 9 Tweak chunking, top_k, prompt template, or model in your code. Run braintrust eval again. Compare new experiment to previous.
Key takeaway: Your local machine is the orchestrator for steps 1-12. It runs Python code that makes API calls outbound to three managed services — OpenAI (compute-heavy embedding and generation), Pinecone (vector storage and search), and Braintrust (trace collection and experiment management). The only steps that run "in the cloud" without your machine involved are the Braintrust UI interactions (13-14).

Diagram 2 — CI/CD Pipeline Workflow
GitHub Actions runner replaces your machine. Same API calls, different orchestrator.
Phase A — Trigger
# Step Runs In Calls Out To What Happens
1 Push code YOUR MACHINE GitHub git push sends updated eval scripts, prompts, or RAG code to the repository.
2 Workflow triggers GITHUB GH Actions Runner GitHub detects the push event, matches the workflow trigger (on: push), and spins up an ephemeral Ubuntu runner.
Phase B — Runner Setup
# Step Runs In Calls Out To What Happens
3 Checkout + install GH ACTIONS RUNNER PyPI, GitHub actions/checkout clones repo. pip install pulls braintrust, autoevals, openai, pinecone-client.
4 Load secrets GH ACTIONS RUNNER GitHub Secrets Injects BRAINTRUST_API_KEY, OPENAI_API_KEY, PINECONE_API_KEY as environment variables.
Phase C — Evaluation (identical to local steps 9-12)
# Step Runs In Calls Out To What Happens
5 braintrust eval GH ACTIONS RUNNER Braintrust API Same CLI command. Pulls dataset, creates experiment. The runner is now the orchestrator instead of your laptop.
6 Execute task × N GH ACTIONS RUNNER OpenAI API
Pinecone API		 For each test case: embed query → retrieve from Pinecone → generate via OpenAI/BT Proxy. Identical API calls as local development.
7 Score outputs GH ACTIONS RUNNER OpenAI API LLM-as-judge scorers call OpenAI. Deterministic scorers run as pure computation on the runner.
8 Publish experiment GH ACTIONS RUNNER Braintrust API Results pushed to Braintrust. Experiment appears in UI alongside your locally-run experiments.
Phase D — Quality Gate
# Step Runs In Calls Out To What Happens
9 Check thresholds GH ACTIONS RUNNER local logic Script checks if avg scores meet minimum thresholds (e.g., Factuality ≥ 0.8). If below threshold, exit 1 fails the pipeline.
10 Pass / Fail GITHUB — GitHub marks the commit check as passed or failed. PR is blocked on failure. Green check on success.
Key takeaway: The CI pipeline is your local workflow with the orchestrator swapped. Instead of your machine running Python and making API calls, the GitHub Actions runner does it. The external service calls (OpenAI, Pinecone, Braintrust) are identical. The only new elements are the trigger mechanism (git push → workflow), secret injection, and the quality gate logic at the end. This is why you develop locally first and CI "just works" — same code, same commands, same API calls.
